# DuaBiskutTelur — Engineering Handover

A single document for an engineer taking over this project: what it is, how it
is built, how to run and deploy it, what is known to be unfinished, and what to
check before a release. User-facing setup lives in [README.md](README.md);
this file goes deeper into architecture, operations, and open items.

> Environment-specific values (server addresses, credentials, signing
> identities) are deliberately **not** in this public document. They live in
> `HANDOVER.private.md`, which is gitignored — obtain it from the project
> owner.

---

## 1. Project overview

DuaBiskutTelur is a mobile-first, installable PWA for meal tracking with a
Malaysian focus. The user photographs a meal (or scans a product barcode); the
system identifies the foods, estimates calories and macros, grades the meal
1–100 (A+–D) with a **deterministic scoring engine** (not AI), and generates
friendly AI feedback. Signed-in users (Google OAuth) get persistent history,
dashboards, achievements, water and weight tracking; anonymous visitors can use
the full analyze flow with session-only state.

## 2. Technology stack

| Layer | Technology |
|---|---|
| Backend | Java 17, Spring Boot 3 (web, security, OAuth2 client, data-jpa), Maven |
| Database | H2 (file mode locally; TCP server mode in production), Flyway migrations |
| AI | Google Gemini — vision (`gemini-flash-latest` + fallbacks) and text feedback (`gemini-flash-lite-latest` + fallbacks) |
| Nutrition data | USDA FoodData Central (photo flow), Open Food Facts (barcode flow) |
| Frontend | React 18, Vite 5, Tailwind CSS 3, vite-plugin-pwa (Workbox) |
| PDF export | OpenPDF |
| Android | Trusted Web Activity generated by Bubblewrap (`android/generate.js`) |
| Deployment | Docker Compose (2 containers), nginx inside the frontend container |

## 3. Architecture summary

```
Browser (PWA)
  │ HTTPS
  ▼
CDN/TLS proxy (e.g. Cloudflare) — terminates TLS, forwards HTTP + X-Forwarded-Proto: https
  │
  ▼
Host nginx on the VM — server_name routing → 127.0.0.1:8081
  │
  ▼
Docker: frontend container (nginx serving the built React app)
  │  /api/*, /oauth2/*, /login/oauth2/* proxied → backend:8080
  │  per-IP rate limiting on the unauthenticated endpoints (realip-aware)
  ▼
Docker: backend container (Spring Boot)
  ├─ H2 file DB (named volume; optional TCP server for SQL tools)
  ├─ Google Gemini API (vision + feedback, keys backend-only)
  ├─ USDA FoodData Central / Open Food Facts
  └─ Google OAuth2 (login optional — visitors allowed)
```

Backend layering is `controller → service → client`. `GeminiClient` implements
the provider-agnostic `VisionAnalysisClient` + `FeedbackClient` interfaces, so
the AI vendor can be swapped without touching the service layer.

### Key design decisions

1. **Deterministic scoring, AI only at the edges.** The grade is pure Java
   arithmetic over resolved nutrition facts (`ScoringService`, thresholds
   tunable in `application.yml` under `scoring.*`). AI identifies foods and
   writes feedback prose — it never decides the score, which also limits the
   blast radius of prompt injection via photos.
2. **Two independent resilience layers for Gemini** (`GeminiClient`):
   key rotation on 429 (per-key cooldowns, primary preferred after cooldown)
   and model fallback on 5xx/timeouts (overload is model-side, so key rotation
   wouldn't help). Exponential backoff only when every key×model fails, then a
   friendly 503 (`ANALYZER_BUSY`).
3. **Login is optional.** Visitors get the full analyze flow; nothing is
   persisted for them (no DB row at all). Signing in adds persistent history,
   profile, and targets.
4. **Nutrition sources are labeled.** Every item carries `source`:
   `usda` (database match), `estimated` (vision-model fallback), or `barcode`
   (label data) — surfaced in the UI as a trust signal.
5. **H2 in TCP-server mode in production** (started from `main()` when
   `H2_TCP_PORT` is set) because `AUTO_SERVER=TRUE` silently rejects real
   remote SQL clients. Local dev uses the plain file URL from
   `application.yml`; production overrides via `SPRING_DATASOURCE_URL`.
6. **Same-origin production, CORS only for dev.** nginx proxies `/api` so the
   browser never makes a cross-origin call in production; the CORS allowlist
   contains only the Vite dev server by default (`app.cors-allowed-origins`).
7. **Explicit camera/gallery chooser** on mobile — the `capture` attribute
   behaves inconsistently across Android/iOS, so the app renders its own
   bottom sheet with two file inputs.
8. **Service worker never intercepts auth or API navigations**
   (`navigateFallbackDenylist` for `/api`, `/oauth2`, `/login/oauth2`,
   `/logout`) — a cached-SPA-served-instead-of-OAuth-redirect bug motivated
   this. App updates apply through a gate (`swUpdateGate.js`) so a deploy
   never reloads the page mid-analysis or mid-scan.
9. **Time zone is server-global.** Every "today"/streak boundary uses
   `ZoneId.systemDefault()`; the container pins `TZ` in `docker-compose.yml`.
   Known limitation for travelling users; acceptable single-market tradeoff.

## 5. Setup (local development)

Prerequisites: Java 17+, Maven, Node 18+.

```bash
# Backend (port 8080) — runs with zero env vars in mock mode:
# no GEMINI key -> realistic mocked analysis; no Google OAuth -> sign-in fails at Google
cd backend && mvn spring-boot:run

# Frontend (port 5173, proxies /api to the backend)
cd frontend && npm install && npm run dev
```

On Windows, if the backend dies at startup with
`Unable to establish loopback connection`, see Troubleshooting below.

## 6. Environment variables

All backend-only; copy `.env.example` and fill in. None are required just to
boot for UI development.

| Variable | Required | Purpose |
|---|---|---|
| `GEMINI_API_KEY` | For real analysis | Primary Gemini key (free tier: aistudio.google.com) |
| `GEMINI_API_KEY_2/_3` | Optional | Backup keys used automatically on 429 |
| `USDA_API_KEY` | Optional | Nutrition lookups; without it, model estimates are used |
| `GOOGLE_CLIENT_ID/SECRET` | For sign-in | OAuth Web client; redirect URI `https://<your-domain>/login/oauth2/code/google` |
| `CORS_ALLOWED_ORIGINS` | Optional | Extra credentialed origins (not needed in the standard setup) |
| `H2_TCP_PORT` / `H2_TCP_BASEDIR` | Production | Start the H2 TCP server for SQL tools |
| `SPRING_DATASOURCE_URL` | Production | Point the app at the TCP server URL |
| `TZ` | Production | The market's time zone — all daily boundaries use it |

## 7. Deployment

`docker-compose.yml` builds two images (multi-stage; backend runs as a
non-root user). Standard sequence on any Docker host:

```bash
# with a filled .env (chmod 600) next to docker-compose.yml
docker compose build
docker compose up -d
```

The frontend container binds to `127.0.0.1:8081` for shared-host setups — a
host-level nginx server block routes the public domain to it. On a dedicated
host, change the binding to `80:80` and skip the host proxy. TLS is expected
to be terminated in front (CDN or host nginx + certbot); the proxy chain must
pass `X-Forwarded-Proto`/`X-Forwarded-For` (the container nginx already
forwards both, and the rate limiter derives the real client IP from them).

After deploying, verify:

```bash
curl -s -o /dev/null -w "%{http_code}\n" https://<your-domain>/        # 200
curl -s https://<your-domain>/api/history                              # 401 JSON
```

**Docker cache gotcha:** frontend rebuilds have been observed reporting every
layer `CACHED` (including `COPY . .`) despite changed files — pass
`--no-cache` for frontend builds, or verify the served asset hash matches the
local `vite build` output.

Nightly DB backups: `ops/backup-db.sh` via cron on the host (see `ops/README.md`).

## 8. Android app (TWA)

The `android/` directory is a Bubblewrap-generated Trusted Web Activity
wrapping the deployed PWA. Only `android/generate.js` is committed — it
regenerates the entire project from the live web manifest (set the manifest
URL at the top of the file to your deployed domain first):

```powershell
cd android
node generate.js                       # regenerates app/, twa-manifest.json, gradle files
.\gradlew.bat assembleRelease bundleRelease   # unsigned outputs
# then zipalign + apksigner (APK) and jarsigner (AAB) with YOUR upload keystore
```

The signing keystore, its password file, and signed artifacts are gitignored
and must never be committed. **Back the keystore up durably — losing it means
the app can never be updated on Play Console.** `frontend/public/.well-known/
assetlinks.json` must carry your keystore's SHA-256 fingerprint and package
name for TWA full-trust mode (this file is public by design).

Note: JDK 17 specifically is required (AGP rejects 21+); always invoke
`gradlew.bat` by absolute path on Windows.

## 9. Testing

```bash
cd backend && mvn test        # full suite: scoring bands, endpoint E2E (MockMvc),
                              # Gemini timeout repro (embedded HTTP server),
                              # cookie-flags invariant (real servlet container)
cd frontend && npm run build  # production build
node scripts/check-i18n-parity.mjs   # en/zh/ms dictionary key parity (also in CI)
```

CI (`.github/workflows/ci.yml`) runs backend tests and the frontend
build+parity check on every push and pull request. There are currently **no
frontend unit tests** — see Outstanding issues.

## 10. Known limitations

- **Single instance by design**: H2 file DB + in-memory sessions. The exit
  path, if ever needed, is Postgres + Spring Session.
- **Server-global time zone** (decision #9) — travelling users' "today" follows
  the server's market time zone.
- **PWA update lag**: after a deploy, already-open clients pick up the new
  build via the gated auto-update; a full close-and-reopen is occasionally
  needed on iOS.
- **Origin serves plain HTTP**; TLS depends entirely on the fronting proxy
  being configured correctly.
- **No monitoring/alerting** on the production host; health is checked
  manually (`docker compose ps` / logs). No actuator endpoints yet.
- The Gemini model-fallback path is unit-tested but has not been observed
  end-to-end against a real, live provider outage.

## 11. Outstanding issues (prioritized)

From the production-readiness and engineering reviews (details in the review
documents / issue tracker):

1. Barcode logging happens on a GET (`/api/barcode/{code}`) — should be a POST
   (HTTP safety; prefetchers can silently log meals).
2. `water_entry` lacks `UNIQUE(user_id, date)` — concurrent first-taps can
   create duplicate rows that 500 the water endpoints for the day. Needs a V3
   migration (+ indexes on all `user_id` query paths, + FKs).
3. `GeminiClient.callWithRetry` has no total wall-clock budget — a slow
   provider can hold request threads for minutes.
4. Open Food Facts outages are reported as "product not found" (404) instead
   of a retryable 503.
5. Frontend: no routing/back-button integration (Android back exits the app);
   no result-correction UX; visitor session history is not migrated into the
   account on first sign-in.
6. No account deletion / data export endpoints (privacy compliance).
7. `FoodKeywords` matches substrings ("steak" contains "tea"), corrupting the
   denormalized beverage/coffee flags for some meals.
8. PDF export is English-only and its Helvetica font cannot render Chinese
   meal content.

## 12. Future roadmap

Near-term: the outstanding list above. Mid-term: manual meal entry/search,
meal-type tags, weekly protein/sodium trends, reminders via PWA push, history
pagination beyond the current 50-entry cap, iOS install guidance and splash

cards, monitoring/observability stack.

## 13. Troubleshooting

- **Backend won't start on Windows** — `Unable to establish loopback
  connection … Invalid argument`: endpoint-security software blocks the JVM's
  unix-domain socket emulation. Run with
  `-Djdk.net.unixdomain.tmpdir=C:\Users\Public` (the Maven surefire profile
  and README document the exact commands).
- **Sign-in "does nothing"** — verify the OAuth redirect URI in Google Cloud
  Console matches `https://<your-domain>/login/oauth2/code/google` exactly,
  and that the service worker denylist is intact (`vite.config.js`).
- **`ANALYZER_BUSY` responses** — all Gemini keys/models are rate-limited or
  down; check backend logs for `cooling down` / `trying next fallback model`.
- **Frontend deploy shows old UI** — Docker layer cache (see §7) or the PWA
  update gate waiting for an idle moment; verify the served asset hash.
- **H2 remote access refused** — the DB must run in TCP-server mode
  (`H2_TCP_PORT`); `AUTO_SERVER=TRUE` cannot accept real remote clients.

## 14. Release checklist

- [ ] `mvn test` green, `npm run build` green, i18n parity green (CI enforces all three)
- [ ] `.env` present on the host, `chmod 600`, never committed
- [ ] OAuth redirect URI registered for the deployment domain
- [ ] DNS + TLS proxy pointing at the host; `X-Forwarded-Proto` verified
      (session cookie must come back `Secure`)
- [ ] `docker compose build --no-cache frontend` if frontend changed
- [ ] Post-deploy smoke: `/` → 200, `/api/history` → 401, one real analyze
- [ ] DB backup cron installed and writing (`ops/backup-db.sh`)
- [ ] Android: keystore + password backed up off-host (see private notes)
- [ ] No secrets in `git status` / staged diff before any push
